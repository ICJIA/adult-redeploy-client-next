# Live Islands — Post-Build CMS Preview for Astro / Tailwind / Alpine (v1)

> **Portable playbook.** Copy this file into any Astro + Tailwind + Alpine site
> that pulls a headless CMS (Strapi, etc.) at build time. It explains — for a
> human or an LLM agent — how to make CMS content that's published **after** the
> last build appear **immediately**, without waiting for a rebuild, and without
> leaving the static-site model.
>
> Authored from the Adult Redeploy Illinois (ARI) build, 2026-06. First shipped
> for `/news/<slug>` detail pages, then extended to `/about/meetings/<cat>/<slug>`
> and the meetings listing. Everything here is framework-portable; only the CMS
> queries and the per-surface markup are site-specific.

---

## 1. The problem (one paragraph)

A static Astro site bakes every page at build time. A CMS detail route like
`/news/[slug]` is generated by `getStaticPaths()` over a **build-time snapshot**
of the CMS. So when an editor publishes a *new* article after the build:

- the homepage/listing **live islands** (if you have them) fetch the latest list
  client-side and show the new item with a link — **but**
- clicking that link hits `/news/<new-slug>`, which has **no static file** →
  the host serves `404.html`.

The page only appears after a rebuild. Editors experience this as "the site is
broken / a day behind," especially when they want to publish → review the live
page → tweak → re-review.

**This playbook closes that gap while staying 100% static** (no SSR adapter, no
serverless functions): the **404 page becomes a client-side live-render host**.

---

## 2. Prerequisite mental model — the three live-CMS render models

| Model | How edits show without a rebuild | When |
|---|---|---|
| **Pure SSG** | They don't (rebuild only) | Low-churn content |
| **Static + live islands** ← *this playbook* | Alpine refetches the CMS in the browser and swaps a region in | Small/medium sites; live edits without server functions; CMS can take direct browser load |
| **Cached SSR + tag-purge** | Server renders per request, edge-cached, purge-on-publish | Large/high-traffic/SEO-critical |

This playbook assumes you're on **static + live islands** and already refresh
*listings* (home cards, index tables) live. Its job is the model's **blind
spot**: a detail page whose **HTML doesn't exist yet** because the slug is new.

**Key distinction (do not conflate):**
- A **listing** island (home/index) live-refreshes an *existing* region → a new
  item *appears in the list*. This is the easy part and probably already works.
- A **detail** page for a new slug has **no page to hydrate** → it 404s. That's
  what the **404-host live-render** below fixes.

---

## 3. The solution in one diagram

```
GET /news/<new-slug>
  → no static file → host serves 404.html  (HTTP 404 — correct & fine)
      404.astro:
        (pre-paint inline script) detect /news/<slug> from location.pathname
            → window.__nf = { surface:'newsDetail', slug }
            → <html data-nf-state="checking">  (hide 404 UI, show "Loading…")
        Alpine mounts liveEntry({ surface, slug, sig:'', bootstrap:true })
            → gqlFetch(CMS, BY_SLUG, { slug })
                 found → applyDetail() paints title/body/… into a shared shell
                         → onHit → data-nf-state="rendered" (show the article)
                 missing/error → onMiss → data-nf-state="default" (real 404 UI)
  later: editor tweaks in the CMS → tab refocus → liveEntry refetches → updates
```

HTTP status stays **404** (the canonical page genuinely doesn't exist yet) — which
is invisible to a human and *correct* for crawlers. The real, SEO-indexed static
page lands on the next rebuild (see §8 backstop), at which point the 404 host is
no longer reached for that slug.

## 4. Components (with portable code)

You need five small pieces. Three are pure + unit-testable; two are markup.

### 4.1 Pure URL→surface matcher (`src/lib/live/fallback/detect.ts`)

Maps a path to the live "surface" + slug. **Pure** (no `window`) so it unit-tests
in a node test runner. The 404 page's inline script mirrors it (it can't import).

```ts
export interface FallbackSurface {
  prefix: string;          // e.g. "/news/" or "/about/meetings/"
  surface: string;         // a liveConfig.entries key, e.g. "newsDetail"
  segments?: number;       // path segments after prefix; slug = LAST. Default 1.
}
export interface FallbackMatch { surface: string; slug: string; }

export const FALLBACK_SURFACES: FallbackSurface[] = [
  { prefix: '/news/', surface: 'newsDetail' },
  { prefix: '/about/meetings/', surface: 'meetingDetail', segments: 2 }, // /<cat>/<slug>
];

export function detectSurface(
  pathname: string, base: string, surfaces = FALLBACK_SURFACES,
): FallbackMatch | null {
  let path = pathname;
  if (base && path.indexOf(base) === 0) path = path.slice(base.length) || '/';
  for (const e of surfaces) {
    if (path.indexOf(e.prefix) !== 0) continue;
    const rest = path.slice(e.prefix.length).replace(/\/+$/, '');
    if (!rest) return null;
    const segs = rest.split('/');
    if (segs.length !== (e.segments ?? 1) || segs.some((s) => !s)) return null;
    return { surface: e.surface, slug: segs[segs.length - 1] };
  }
  return null;
}
```

### 4.2 A pure "bootstrap effect" reducer (`behavior/bootstrap.ts`)

The bootstrap render has no build baseline, so the host must know when the first
paint succeeded (hide "Loading…") vs. when there's nothing to show (revert to
404). Pure → node-testable; `liveEntry` calls it and fires the host's callbacks.

```ts
export type RefreshOutcome = 'view' | 'null' | 'error';
export function bootstrapEffect(
  bootstrap: boolean | undefined, rendered: boolean, outcome: RefreshOutcome,
): 'hit' | 'miss' | 'none' {
  if (!bootstrap || rendered) return 'none';   // not bootstrap, or already painted
  return outcome === 'view' ? 'hit' : 'miss';
}
```

### 4.3 `liveEntry` bootstrap mode (the generic Alpine factory)

Your existing single-record island already fetches by slug, signature-gates, and
swaps. Add a **bootstrap** path: `sig:''` makes the first fetch always apply, and
`onHit`/`onMiss` tell the host the outcome. Stays generic (no per-site specifics).

```ts
return (opts: { surface: string; slug: string; sig: string;
                bootstrap?: boolean; onHit?: () => void; onMiss?: () => void }) => ({
  _sig: opts.sig, _rendered: false, /* …controllers… */
  async refresh() {
    const surface = config.entries[opts.surface];
    if (!surface) return;                 // unregistered surface → no-op (key trick!)
    /* …abort prior, progress.start()… */
    try {
      const data = await gqlFetch(config.ctx.endpoint, surface.query,
                                  surface.variables(opts.slug), { signal });
      const view = surface.select(data, config.ctx);
      if (!view) {
        if (bootstrapEffect(opts.bootstrap, this._rendered, 'null') === 'miss') opts.onMiss?.();
        return;
      }
      const sig = surface.signature(view);
      if (sig === this._sig) return;       // unchanged → no swap
      this._sig = sig;
      await surface.applyTo(this.$root, view, config.ctx);
      if (bootstrapEffect(opts.bootstrap, this._rendered, 'view') === 'hit') {
        this._rendered = true; opts.onHit?.();
      }
    } catch {
      if (bootstrapEffect(opts.bootstrap, this._rendered, 'error') === 'miss') opts.onMiss?.();
    } finally { /* progress.done() */ }
  },
});
```

### 4.4 A shared "detail shell" component per surface (`*DetailShell.astro`)

ONE component renders the detail markup for BOTH the static page (item provided →
filled) AND the 404 host (no item → empty `data-live` hooks the island fills).
This guarantees static and live can't drift.

```astro
---
interface Props { item?: Item | null; }
const { item = null } = Astro.props;
const ctx = { endpoint: '', basePath: import.meta.env.BASE_URL || '/base' };
---
<ArticleCard title={item?.title ?? ''} badge="…">
  <p data-live="posted">{item ? `Posted: ${formatDate(item.date)}` : ''}</p>
  {item ? <Markdown content={item.content} dataLive="body" />
        : <div class="prose max-w-none" data-live="body"></div>}
  <div data-live="tags" set:html={item ? renderTags(item.tags, ctx) : ''}></div>
</ArticleCard>
```

The real detail page becomes `<div x-data="liveEntry({surface, slug, sig})"><XDetailShell item={item} /></div>`.

### 4.5 The 404 host (`404.astro`) — one sub-host per surface

```astro
---
import { FALLBACK_SURFACES } from '../lib/live/fallback/detect';
const base = import.meta.env.BASE_URL || '/base';
---
<BaseLayout title="Page not found">
  {/* Pre-paint detect — MIRRORS detectSurface(); keep in sync. */}
  <script is:inline define:vars={{ base, map: FALLBACK_SURFACES }}>
    (function () {
      window.__nfState = (s) => document.documentElement.setAttribute('data-nf-state', s);
      var path = location.pathname;
      if (base && path.indexOf(base) === 0) path = path.slice(base.length) || '/';
      for (var i = 0; i < map.length; i++) {
        var e = map[i];
        if (path.indexOf(e.prefix) !== 0) continue;
        var rest = path.slice(e.prefix.length).replace(/\/+$/, '');
        if (!rest) break;
        var segs = rest.split('/'), bad = segs.length !== (e.segments || 1);
        for (var j = 0; j < segs.length; j++) if (!segs[j]) bad = true;
        if (bad) break;
        window.__nf = { surface: e.surface, slug: segs[segs.length - 1] };
        document.documentElement.setAttribute('data-nf-surface', e.surface);
        window.__nfState('checking');
        window.setTimeout(function () {                       // safety net
          if (document.documentElement.getAttribute('data-nf-state') === 'checking')
            window.__nfState('default');
        }, 8000);
        break;
      }
    })();
  </script>

  <div class="nf-live">
    <p class="nf-loading">Loading…</p>
    {/* One host per surface. Only the matching surface fetches; the others get
        surface:undefined → liveEntry.init no-ops. applyTo is scoped to each
        host's own $root, so the shells' shared data-live hooks never collide. */}
    <div class="nf-shell" data-nf-shell="newsDetail"
         x-data="liveEntry({ surface: (window.__nf && window.__nf.surface==='newsDetail') ? 'newsDetail' : undefined, slug: window.__nf && window.__nf.slug, sig:'', bootstrap:true, onHit:()=>window.__nfState('rendered'), onMiss:()=>window.__nfState('default') })">
      <NewsDetailShell />
    </div>
    <div class="nf-shell" data-nf-shell="meetingDetail"
         x-data="liveEntry({ surface: (window.__nf && window.__nf.surface==='meetingDetail') ? 'meetingDetail' : undefined, slug: window.__nf && window.__nf.slug, sig:'', bootstrap:true, onHit:()=>window.__nfState('rendered'), onMiss:()=>window.__nfState('default') })">
      <MeetingDetailShell />
    </div>
  </div>

  <section class="nf-fallback"> … your normal 404 UI … </section>
</BaseLayout>

<style>
  .nf-live, .nf-shell { display: none; }
  :root[data-nf-state="checking"] .nf-fallback,
  :root[data-nf-state="rendered"] .nf-fallback { display: none; }
  :root[data-nf-state="checking"] .nf-live,
  :root[data-nf-state="rendered"] .nf-live { display: block; }
  :root[data-nf-state="rendered"] .nf-loading { display: none; }
  /* On render, reveal ONLY the shell whose surface matched. */
  :root[data-nf-state="rendered"][data-nf-surface="newsDetail"] .nf-shell[data-nf-shell="newsDetail"],
  :root[data-nf-state="rendered"][data-nf-surface="meetingDetail"] .nf-shell[data-nf-shell="meetingDetail"] { display: block; }
</style>
```

**Why per-surface sub-hosts work without colliding:** each `x-data` is its own
Alpine component, so `this.$root` (what `applyTo` searches with
`root.querySelector('[data-live=…]')`) is that host's subtree. Two shells can both
use `data-live="title"`/`"body"` and never clash. Only the surface-matched host
gets a real surface and fetches; the others no-op via `if (!config.entries[surface]) return`.

## 5. Gotchas — the expensive lessons (read before shipping)

1. **CMS by-slug filter that silently returns nothing.** Strapi v3 GraphQL
   ignores a variable used *inside* a `where` object literal
   (`where:{ slug:$slug }` → returns ALL rows or none). Pass the whole filter as a
   JSON variable: `query($where: JSON){ posts(where:$where) }` with
   `variables:{ where:{ slug } }`. **Verify the by-slug query actually returns the
   record** (curl the endpoint) before blaming the island — a broken by-slug query
   is silently masked on built pages (the island just keeps the static DOM) and
   only surfaces on the 404 host (no static DOM → "miss" → 404).

2. **Trailing/leading whitespace in slugs.** A CMS slug like `"my-post "` can't
   round-trip through a URL, so it breaks the static route, the links, AND the
   exact-match by-slug lookup. **Trim every slug** at the build fetch (normalize
   the snapshot) AND in any client-side link builder. Add a unit-tested
   `cleanSlug = (s) => typeof s === 'string' ? s.trim() : s`.

3. **Date "off by a day" (timezone).** Date-only CMS values stored as midnight
   UTC (`2026-06-05T00:00:00Z`) render as the *previous day* for viewers behind
   UTC if you format with the viewer's local zone. If the CMS field is
   semantically **date-only**, format by the **stored Y-M-D part** (no `Date`
   conversion) so "June 5" always reads "June 5". Apply the same in the SSR
   formatter AND the client `x-for` table formatter so static + live agree.
   ```ts
   const m = /^(\d{4})-(\d{2})-(\d{2})/.exec(iso);
   if (m) return `${MONTHS[+m[2]-1]} ${+m[3]}, ${m[1]}`;
   ```
   For "is it upcoming/today," compare the **date part** against today's date in
   the org's timezone (`new Date().toLocaleDateString('en-CA',{timeZone:'America/Chicago'})`).

4. **Based-path sites behind a path-stripping proxy.** If the site ships
   `base:'/x'` and is proxied at `host/x/*` with the prefix **stripped** before it
   reaches the site, then `location.pathname` is `/news/<slug>` under the proxy but
   `/x/news/<slug>` on the raw deploy host. **Strip an optional leading `base`**
   in detection (the snippet above does). Netlify `_redirects`/`netlify.toml`
   `from` must be **un-prefixed**; `to` keeps the prefix.

5. **CSP.** The inline detect script needs `script-src 'unsafe-inline'` (or a
   hash), Alpine needs `'unsafe-eval'`, and the browser CMS fetch needs the API
   origin in `connect-src`. If your listings already live-fetch, `connect-src` is
   already correct — **zero header change**. If you hash inline scripts, add the
   detect script's hash.

6. **Pure-vs-browser module split.** The `.astro` build frontmatter must not
   import browser-only modules (anything touching `window`/`document` at module
   top level) — importing *any* symbol evaluates the whole import graph. Keep
   pure helpers (signature, slug, date, detect, render-to-string) in modules with
   no browser deps; keep the Alpine factory (which pulls `progress`/`announce`/
   transition) in a separate file imported only by the client entrypoint.

7. **No-flash.** Set `data-nf-state` from the **pre-paint inline script** (runs
   before Alpine's deferred module), so the visitor never sees the 404 flash
   before the article. Always include an ~8s safety net that reverts to the real
   404 if the island never resolves.

8. **Shared-`.mjs` helper for build + browser.** The build script (Node, `.mjs`)
   and the TS render layer both need some helpers (slug trim, category maps). Put
   them in a plain-ESM `.mjs` with a `.d.mts` type file and import from both.

## 6. Testing strategy (matches a node-env test runner)

- **Unit (pure, fast):** `detectSurface` (both hosts, all reject cases),
  `bootstrapEffect` (hit/miss/none gating), `cleanSlug` + `normalizeSlugs`, the
  date formatter (date-part, no TZ shift), `selectUpcoming` (today-or-later by
  date), and every render-to-string fn (escaping, hrefs).
- **DON'T** unit-test the DOM-applying functions or `liveEntry.refresh` if your
  runner is node-only — they're `window`/DOM-coupled. Verify those in a real
  browser on a **deploy preview** (never `astro dev`, which ships unminified JS +
  the toolbar and skews Lighthouse/behavior).
- **Browser acceptance (the author loop):** publish a record *after* the build →
  visit its detail URL → it renders (no 404) → edit in CMS → refocus tab → it
  updates → a bogus slug → reverts to the normal 404 UI → axe AA = 0 on the
  rendered state.

## 7. Step-by-step: adopt on a new site

1. Confirm you're **static + live islands** (listings already live-refresh). If
   not, build that first; this extends it.
2. Confirm the CMS exposes a **single-record by-slug** query that actually returns
   the record (see gotcha #1). Reuse your existing detail-refresh query.
3. Add `detect.ts` (§4.1) with one `FALLBACK_SURFACES` row per detail route.
4. Add `bootstrap.ts` (§4.2) and wire `bootstrap`/`onHit`/`onMiss` into your
   `liveEntry` factory (§4.3).
5. Extract a `*DetailShell.astro` per surface (§4.4); point both the real detail
   page and the 404 host at it.
6. Rebuild `404.astro` as the multi-surface host (§4.5) + the `[data-nf-state]` /
   `[data-nf-surface]` CSS state machine.
7. Apply the gotchas: trim slugs (#2), date-only format (#3), base-strip (#4),
   confirm CSP (#5).
8. Add unit tests (§6). Verify on a deploy preview (§6).
9. Wire the backstop (§8).

To add **another** surface later: one `FALLBACK_SURFACES` row + one
`*DetailShell` + one sub-host in `404.astro`. The query/renderer already exist if
that surface has a live detail refresh.

---

## 8. Backstop: rebuild-on-publish (do this too)

The 404 host is the **pre-rebuild bridge** (instant, but HTTP 404, not indexed).
The canonical artifact is still the static page. So also wire:

- A **build hook** on the host (Netlify: Site config → Build hooks).
- A **CMS webhook** on create/update/publish that POSTs the build hook URL.

Then a new record's real, SEO-indexed static page lands within ~1–3 min, and the
404 host only ever serves the brief window in between. (Guard/debounce the hook
later if needed.)

---

## 9. Acceptance criteria

1. A record published after the build renders at its detail URL with no manual
   rebuild — on the proxied host AND the raw deploy host.
2. Editing it and refocusing the tab updates the rendered page (author loop).
3. A nonexistent slug shows the normal 404 UI at 404 status (no perpetual
   spinner; an 8s safety net guarantees it).
4. Every pre-built page is unchanged; listings still live-refresh.
5. Dates show as entered (no day-off); "upcoming" includes today.
6. Unit tests green; production build green; axe AA = 0 on the rendered state.
7. No CSP/header change beyond an inline-script allowance (if hashing) — the CMS
   origin was already in `connect-src`.
8. Backstop rebuild-on-publish replaces the live fallback with the canonical
   static page within minutes.
